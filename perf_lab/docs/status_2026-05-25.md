## Chunked CE Prototype

Added:

- `perf_lab/auralis_perf_kernels/chunked_ce.py`
- `chunked_linear_cross_entropy(hidden, weight, labels, chunk_size=8192)`
- C++ extension backend: `perf_lab/csrc/chunked_ce_ext.cpp`
- benchmark integration in `perf_lab/benchmarks/bench_vocab_ce.py`
- chunk sweep benchmark: `perf_lab/benchmarks/sweep_chunked_ce.py`

The prototype is exact in fp32 and avoids materializing the full logits tensor.
It has two backends:

- `impl=python`: Python loops plus Torch GEMMs per vocab chunk.
- `impl=cpp`: C++ extension loops plus Torch GEMMs per vocab chunk.

The C++ backend removes Python from the loop but still launches multiple GEMMs,
so it is a bridge to the final CUDA kernel rather than the final speed path.

### Chunk-Size Sweep

Interpretation:

- fp32 parity is clean enough to trust the algorithm.
- bf16 full PyTorch CE reports rounded loss values; chunked CE computes logsumexp
  in fp32 over bf16 matmul chunks, so loss can differ slightly while gradients
  stay close.
- Memory saving becomes meaningful as token count grows.
- Speed is currently worse because the algorithm still launches many GEMMs.
  C++ did not change this much at Auralis shape, which proves launch/GEMM
  structure is the real target.
- A true CUDA/Triton kernel needs to reduce launches and avoid storing logits.
  The validated algorithm is now ready for that implementation.

## Triton Linear-CE Forward Prototype

Added:

- `perf_lab/auralis_perf_kernels/triton_ce.py`
- `perf_lab/benchmarks/bench_triton_ce_forward.py`

The first block kernel needed `tl.dot(..., input_precision="ieee")`; without
that, fp32 loss parity drifted by about 0.02 because Triton used a lower
precision dot path. With the explicit precision setting, fp32 parity is clean.

  two-stage partial reduction only if the partial buffer can be bounded.
- Current best tile family remains around `block_m=32`, `block_v=32`,
  `block_d=64` for mid-size shapes.

## Bounded Partial Weight Reduction and Low-Memory Atomics

Added:

- `triton_linear_ce_backward_partial_weight(...)`
- `triton_fused_linear_cross_entropy(..., backward_mode="partial_weight")`
- `triton_linear_ce_backward_atomic_lowmem(...)`
- `triton_fused_linear_cross_entropy(..., backward_mode="atomic_lowmem")`
- `--row-group-blocks` benchmark option for bounded partial-weight tests.

`partial_weight` computes `grad_weight` into `(row_group, vocab, dim)` fp32
partials, then reduces those partials into the final weight gradient. This
removes `grad_weight` atomics, but only inside a bounded row group so the
partial buffer does not become unbounded. `atomic_lowmem` keeps the original
atomic work split but allocates `grad_hidden` and `grad_weight` in the input
tensor dtype instead of fp32.

## Update 2026-05-31 (later) — Self-calibrating gate + the scale reversal

Built `perf_lab/benchmarks/ce_promotion_gate.py`: one command that measures the
bf16 noise floor (PyTorch-bf16 vs fp32 ground truth) and each candidate in the
**same** run, derives the pass bar from the *measured* floor (`floor *
--tolerance`, default 1.5x), and prints a PASS/FAIL table on the
promotion-deciding metric (max upstream-gradient L2-rel drift) plus a flat-drift
slope check. Each candidate runs as a **subprocess** so every run gets a fresh
CUDA context — no cross-run allocator state contaminating the next measurement.

This tool immediately corrected a premature conclusion.

### Small shape (vocab=8192) — both pass, drift is NOT the differentiator

```
candidate                   grad_l2_rel  x_floor    loss_abs   max_slope  verdict
pytorch[floor]                 0.003652     1.00     0.03123    7.56e-06     PASS
liger                          0.003596     0.98   0.0002069    6.01e-06     PASS
triton_fused:atomic_mixed      0.001726     0.47   0.0001068    2.75e-06     PASS
```

Key insight the floor-relative view exposes: the ~0.36% drift that earlier work
treated as a *problem* is, at small shape, simply **the bf16 noise floor itself**
(0.0037). No bf16 candidate can beat it, and both candidates sit at/below it.

### Auralis shape (vocab=200000) — the verdict FLIPS

```
candidate                   grad_l2_rel  x_floor    loss_abs   max_slope  verdict
pytorch[floor]                 0.003667     1.00     0.03054    9.75e-06     PASS
liger                           0.01486     4.05   0.0006371    1.13e-05     FAIL
triton_fused:atomic_mixed      0.001265     0.34   0.0001945    9.55e-06     PASS
```

At the real vocab size the result **reverses**: Liger FAILs (4.05x floor) and the
hand-rolled `atomic_mixed` PASSes (0.34x floor — 3x better than even the bf16
floor).

Root cause, isolated (identical under `--accum-dtype fp32` and `none`, so not a
config mistake):

- Liger `head_grad_l2_rel` (LM-head weight) = **0.00126** — below floor. Its
  GEMM-accumulated weight gradient really is excellent.
- Liger `proj_grad_l2_rel` (upstream `grad_hidden`) = **0.01486** — 4x floor.
  Liger computes `grad_hidden = dlogits @ weight`; at vocab=200000 the `dlogits`
  entries are tiny (~softmax 1/200000) and Liger accumulates them through bf16,
  losing precision. `accum_dtype` only governs the *weight* accumulation, so it
  does not help `grad_hidden`.
- `atomic_mixed` keeps `grad_hidden` accumulation in fp32 (the "mixed" part) and
  recomputes logits/softmax in fp32 inside the Triton kernel, so its upstream
  gradient is *more* accurate than a plain bf16 cuBLAS GEMM — hence below floor.

### Corrected verdict

- The hand-rolled `atomic_mixed` investigation was **not** wasted: its fp32
  internal logit/softmax math gives it a real, measurable correctness edge at
  200k vocab that Liger's out-of-the-box backward does not match.
- **Liger is not promotable for our CE as-is** — its upstream-gradient drift is
  4x the bf16 floor at our vocab size. (Liger's RMSNorm/RoPE/SwiGLU are still
  worth evaluating separately; this finding is specific to FLCE's `grad_hidden`.)
- `atomic_mixed` is the current **correctness** leader at 200k (passes the
  floor-relative gate). Its remaining gates before promotion are (a) speed — it
  was slow under contention; re-measure on a free GPU — and (b) atomicAdd
  run-to-run **nondeterminism**, which matters for reproducibility independent of
  drift magnitude.
- The real durable win from this round is the **gate itself**: drift is now a
  one-command, floor-relative, scale-aware PASS/FAIL instead of a pile of raw
  numbers needing interpretation.

### atomic_mixed nondeterminism — quantified, negligible

Two identical 200k runs (same seed, `atomic_mixed`):

| metric | run 1 | run 2 | delta | delta / bf16 floor |
| --- | ---: | ---: | ---: | ---: |
| `proj_grad_l2_rel` | 0.0011569 | 0.0011597 | 2.9e-6 | ~0.08% |
| `head_grad_l2_rel` | 0.0004971 | 0.0005062 | 9.1e-6 | ~0.25% |

The atomicAdd ordering does make the gradient vary run-to-run, but the variation
is **~300-400x smaller than the bf16 noise floor** (0.00367). bf16 training is
already nondeterministic on a larger scale than this, so the atomic
nondeterminism is not a promotion blocker. `atomic_mixed` is therefore **fully
clear on correctness** at 200k — the only remaining gate is speed.

| --- | ---: | ---: | ---: |
| custom (CUDA) | 0.125 | 0.03125 | 1.0294 |
| liger (Triton) | 0.125 | 0.03125 | 1.0022 |

Both match the PyTorch reference at bf16-rounding level. Unlike FLCE, RMSNorm has
no huge-vocab tiny-gradient term, so **Liger RMSNorm is numerically fine for us**.

But the speed numbers this round are **invalid**, and this run proves it directly:
the *identical* PyTorch reference op timed `0.00126s` avg in the custom run and
`0.00286s` avg in the Liger run — a 2.3x swing on the same code. That is the
foundation training run sharing the GPU. So every wall-clock/speedup number from
this session (FLCE and RMSNorm) is contention-bound and must be re-measured on a
free GPU before any speed claim.

## Update 2026-05-31 (clean GPU) — speed gates resolved on a free RTX 3090

The foundation run owns the BITBASTION GPU, so the speed numbers above were all
contention-bound. Ran the gates on a **free local RTX 3090 (sm_86)** via WSL2 +
the `auralis3090` venv (`pip install --no-deps liger-kernel`; perf_lab copied to
`~/perf_lab`). Caveats: (1) 3090 != Blackwell, so *absolute* numbers differ, but
the *scaling* behaviour below is algorithm-driven and architecture-independent;
(2) the 3090's older Triton lacks bf16 `atomic_add`, so `atomic_mixed` could not
run — tested the fp32-buffer `atomic` mode instead (dead on speed regardless).

### CE speed/memory, 200k vocab, bf16 (clean, no contention)

| impl | tokens | time | peak mem | vs full-logits |
| --- | ---: | ---: | ---: | --- |
| full-logits (PyTorch) | 2048 | 0.051s | 5.06 GB | baseline |
| fused `atomic` (hand-rolled) | 2048 | 5.91s | 4.16 GB | **~117x slower**, -0.9 GB |
| liger | 2048 | 0.943s | 5.17 GB | **~18x slower**, +0.1 GB |
| full-logits (PyTorch) | 8192 | 0.200s | 10.38 GB | baseline |
| liger | 8192 | 0.971s | 3.15 GB | ~4.85x slower, **-7.2 GB** |

Parity stayed exact at 200k (`grad_hidden_max_abs` ~7e-7, `grad_weight` ~8e-6).

The crossover is structural: full-logits memory is O(tokens x vocab) and grows
linearly, while liger's memory is ~flat (dominated by the weight gradient) and
its time barely moves (0.94s -> 0.97s as tokens 4x). So full-logits wins on small
token counts; fused CE only wins memory once the logits tensor dominates.

### CE verdict — keep PyTorch full-logits; do NOT promote any fused CE

Our production micro-batch is ~2048 tokens per CE call (seq 2048). At that size
full-logits is **18-117x faster** and uses about the same memory. Fused CE only
pays off at >=~8192 tokens/call, where you trade ~5x slower CE for ~3x less
memory — a deal worth taking only if CE memory is the binding constraint (it is
not today: full CE peak ~5 GB inside a 15-21 GB run).

Consequences:

- The hand-rolled `triton_fused` CE line is **closed**: it is ~100x slower than
  full-logits on a clean GPU. Its correctness edge (fp32 `grad_hidden`, below the
  bf16 floor) is real but moot — we will not promote a 100x-slower kernel.
- Liger FLCE is also not promotable for our config: ~18x slower at our token
  count, and its upstream-gradient drift is 4x the bf16 floor at 200k anyway.
- Promote nothing for CE. Revisit only if batch x seq grows enough that
  full-logits CE memory becomes the OOM wall — then Liger FLCE (memory) is the
  lever, accepting its grad_hidden drift.

### RMSNorm verdict — Liger is a real, clean win

| impl | rows x dim | speedup vs PyTorch ref | parity |
| --- | --- | ---: | --- |
| liger (Triton) | 8192 x 1280 bf16 | **3.13x** | bf16-clean |
| custom (CUDA) | — | could not build (no ninja/nvcc in WSL) | n/a |

Liger RMSNorm is **3.13x faster** than the reference on a clean 3090 and
numerically correct. The hand-rolled CUDA kernel needs a build toolchain and
historically showed only ~1.1-1.3x. **Liger RMSNorm is the recommended kernel**
(fast, correct, maintained, zero build step). Confirm with a clean Blackwell
head-to-head vs `custom` before swapping into `src/auralis`, then also benchmark
Liger RoPE/SwiGLU.

### Bottom line for this whole investigation

- **CE**: stop. Full-logits PyTorch CE is the right choice at our scale. Both the
  hand-rolled fused kernels and Liger FLCE lose on speed at ~2048 tokens/call.
- **RMSNorm**: Liger is the highest-value, lowest-risk perf win surfaced here —
  pursue it (plus RoPE/SwiGLU) once the foundation run frees the Blackwell GPU
  for a confirming head-to-head.
- The durable artifact is the **`ce_promotion_gate.py`** + the clean-GPU
  methodology (WSL2 + `auralis3090` venv + `~/perf_lab`), reusable for the next
  kernel decision without waiting for the production GPU.
