## Chunked CE Prototype

Added:

- `perf_lab/auralis_perf_kernels/chunked_ce.py`
- `chunked_linear_cross_entropy(hidden, weight, labels, chunk_size=8192)`
- C++ extension backend: `perf_lab/csrc/chunked_ce_ext.cpp`
- benchmark integration in `perf_lab/benchmarks/bench_vocab_ce.py`
- chunk sweep benchmark: `perf_lab/benchmarks/sweep_chunked_ce.py`

The prototype is exact in fp32 and avoids materializing the full logits tensor.
It has two backends:

- `impl=python`: Python loops plus Torch GEMMs per vocab chunk.
- `impl=cpp`: C++ extension loops plus Torch GEMMs per vocab chunk.

The C++ backend removes Python from the loop but still launches multiple GEMMs,
so it is a bridge to the final CUDA kernel rather than the final speed path.

### Chunk-Size Sweep

Interpretation:

- fp32 parity is clean enough to trust the algorithm.
- bf16 full PyTorch CE reports rounded loss values; chunked CE computes logsumexp
  in fp32 over bf16 matmul chunks, so loss can differ slightly while gradients
  stay close.
- Memory saving becomes meaningful as token count grows.
- Speed is currently worse because the algorithm still launches many GEMMs.
  C++ did not change this much at Auralis shape, which proves launch/GEMM
  structure is the real target.
- A true CUDA/Triton kernel needs to reduce launches and avoid storing logits.
  The validated algorithm is now ready for that implementation.

## Triton Linear-CE Forward Prototype

Added:

- `perf_lab/auralis_perf_kernels/triton_ce.py`
- `perf_lab/benchmarks/bench_triton_ce_forward.py`

The first block kernel needed `tl.dot(..., input_precision="ieee")`; without
that, fp32 loss parity drifted by about 0.02 because Triton used a lower
precision dot path. With the explicit precision setting, fp32 parity is clean.

  two-stage partial reduction only if the partial buffer can be bounded.
- Current best tile family remains around `block_m=32`, `block_v=32`,
  `block_d=64` for mid-size shapes.

## Bounded Partial Weight Reduction and Low-Memory Atomics

Added:

- `triton_linear_ce_backward_partial_weight(...)`
- `triton_fused_linear_cross_entropy(..., backward_mode="partial_weight")`
- `triton_linear_ce_backward_atomic_lowmem(...)`
- `triton_fused_linear_cross_entropy(..., backward_mode="atomic_lowmem")`
- `--row-group-blocks` benchmark option for bounded partial-weight tests.

`partial_weight` computes `grad_weight` into `(row_group, vocab, dim)` fp32
partials, then reduces those partials into the final weight gradient. This
removes `grad_weight` atomics, but only inside a bounded row group so the
partial buffer does not become unbounded. `atomic_lowmem` keeps the original
atomic work split but allocates `grad_hidden` and `grad_weight` in the input
tensor dtype instead of fp32.

## Loss-Drift Gate

Added:

- `perf_lab/benchmarks/loss_drift_ce.py`

This is a promotion gate for CE candidates. It runs two identical synthetic
optimizer loops:

- reference: PyTorch full logits + `F.cross_entropy`
- candidate: selected chunked/Triton CE implementation

Both loops start from identical toy projection/head weights, consume identical
synthetic batches, and update with the same optimizer. The report tracks loss
drift, gradient drift, and parameter drift across multiple steps. L2-relative
metrics are included because max-relative drift is dominated by near-zero
gradient/parameter entries.

### Smoke Test: fp32 Atomic

Shape: `steps=5`, `tokens=32`, `input_dim=64`, `d_model=64`,
`vocab_size=512`, `fp32`, `adamw`, `triton_fused`, `backward_mode=atomic`.

```json
{
  "loss_abs_max": 1.049041748046875e-05,
  "proj_param_l2_rel_max": 0.002257763408124447,
  "head_param_l2_rel_max": 0.017433855682611465,
  "proj_grad_max_abs": 1.8406659364700317e-05,
  "head_grad_max_abs": 2.1230429410934448e-05,
  "finite": true
}
```

This validates the test harness and shows no instability over a few optimizer
steps. The high max-relative values come from tiny near-zero entries; absolute
errors remain small.

## Mixed Atomic Drift Reduction

Added:

- `triton_linear_ce_backward_atomic_mixed(...)`
- `triton_fused_linear_cross_entropy(..., backward_mode="atomic_mixed")`
- `_bf16match` suffix for `atomic`, `atomic_lowmem`, and `atomic_mixed`

`atomic_mixed` keeps `grad_hidden` accumulation in fp32 but keeps the huge
`grad_weight` buffer in the parameter dtype. This targets the main drift source
found in the previous loss-drift gate while preserving most of the memory win
from `atomic_lowmem`.

The `_bf16match` suffix additionally rounds Triton chunk logits to bf16 before
CE math. This was tested because the PyTorch bf16 reference materializes bf16
full logits before `cross_entropy`.

### 200k Speed/Memory Snapshot

Shape: `tokens=512`, `d_model=1280`, `vocab_size=200000`, `bf16`,
`block_m=32`, `block_v=32`, `block_d=64`, chunked-only benchmark.

| mode | seconds_avg | peak_alloc_gb |
| --- | ---: | ---: |
| `atomic` | 0.03264 | 2.05325 |
| `atomic_lowmem` | 0.02763 | 1.02663 |
| `atomic_mixed` | 0.02917 | 1.02925 |
| `atomic_mixed_bf16match` | 0.02917 | 1.02925 |

Conclusion:

- Exact 0% drift was not reached against PyTorch bf16 full-logits CE.
- `atomic_mixed` is the best current compromise: it removes most of the harmful
  `atomic_lowmem` drift while preserving nearly all of the memory savings.
- The remaining `~0.36%` upstream-gradient L2 drift is likely from Triton/PyTorch
  reduction-order and bf16 matmul/CE semantic differences. That is small, but
  still not a promotion-grade zero-drift result.
- For a production path, the next useful experiment is a PyTorch-compatible
  reference mode that intentionally mirrors full-logits bf16 semantics end to
  end, or a custom kernel with deterministic row-owned reductions.
